## 💡 关键建议

1. **采用"设计先行"路线** (而不是快速蛮干)
   - Week 0 花 7 小时进行架构评审
   - 比之后花 100+ 小时返工要划算

2. **优先完成 events-service**
   - 这是其他所有服务的基础
   - 做得不对就会导致数据一致性问题
   - 建议使用 PostgreSQL Outbox 模式

3. **搜索和推荐不需要过度优化**
   - PostgreSQL FTS 足够 Phase 1
   - Elasticsearch/Milvus 留给 Phase 2
   - ONNX 模型留给 Week 5+

4. **建立统一的事件流**
   - 7 个独立系统 → 1 个统一事件流
   - 解决 reactions/likes 定义不统一的问题
   - 节省 200+ 小时的重复代码和 bug 修复
